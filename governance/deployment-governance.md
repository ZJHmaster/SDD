# 基于风险的部署分级治理 (Deployment Governance)

## 1. 背景
在企业级项目中，部署并非一个简单的技术操作，而是面临严格的权限管控与审批约束。"一刀切"的部署权限管理无法同时满足效率与安全的平衡。本规范通过引入**基于风险的分级策略**，让低风险变更快速通过，高风险变更严格把关。

## 2. 部署风险评级模型

### 2.1 风险因子

| 因子 | 权重 | 低风险 (1分) | 中风险 (3分) | 高风险 (5分) |
|------|:---:|-------------|------------|-------------|
| **变更范围** | 20% | 仅修改配置/文案 | 修改业务逻辑 | 修改核心交易/资金逻辑 |
| **数据库变更** | 20% | 无 DB 变更 | 新增列/索引 | 修改/删除列、数据迁移 |
| **影响服务数** | 15% | 单服务 | 2-3 个服务 | 4+ 个服务或全链路 |
| **API 契约变更** | 15% | 向后兼容 | 新增端点 | 破坏性变更 |
| **用户影响面** | 15% | 内部用户 | 部分外部用户 | 全量外部用户 |
| **可回退性** | 15% | 秒级回退 | 分钟级回退 | 需数据修复，难以回退 |

### 2.2 风险等级计算
```
风险分 = Σ(因子得分 × 权重)
```

| 风险分范围 | 等级 | 标识 |
|:---------:|:---:|:---:|
| 1.0 - 1.8 | 低风险 | 🟢 |
| 1.9 - 3.2 | 中风险 | 🟡 |
| 3.3 - 5.0 | 高风险 | 🔴 |

## 3. 分级审批流程

### 🟢 低风险部署
```
开发者 → CI 管线全通过 → 自动部署到 staging → 冒烟测试通过 → Tech Lead 单人审批 → 生产部署
```
- **审批人数**：1 人 (Tech Lead)
- **审批时效要求**：4 小时内
- **部署窗口**：工作日 10:00-18:00

### 🟡 中风险部署
```
开发者 → CI 管线全通过 → 影响分析报告 → staging 验收 → Tech Lead + 部门主管双人审批 → 灰度发布 → 观察期 → 全量
```
- **审批人数**：2 人 (Tech Lead + 部门技术负责人)
- **前置材料**：变更影响分析报告 (来自 `impact-analysis` 技能)
- **部署策略**：灰度发布（先 10%→30%→100%）
- **观察期**：每阶段至少 30 分钟

### 🔴 高风险部署
```
开发者 → CI 管线全通过 → 影响分析报告 → staging + pre-prod 验收 → Tech Lead + 部门主管 + 架构委员会 三级审批 → 蓝绿/金丝雀部署 → 影子流量验证 → 逐步切量 → 观察期
```
- **审批人数**：3+ 人 (含架构委员会成员)
- **前置材料**：
  - 变更影响分析报告
  - 回退预案文档
  - 关联的 BRS 需求清单
  - 数据库 Migration 的 UP/DOWN 验证截图
- **部署策略**：蓝绿 或 金丝雀（≤5%→10%→30%→100%）
- **观察期**：每阶段至少 1 小时
- **部署窗口**：需提前预约审批

## 4. AI Agent 辅助能力

### 4.1 自动风险评分
当开发者准备提交部署申请时，AI Agent 可自动：
- 分析本次变更的代码 Diff
- 检查是否包含 Migration 文件
- 统计受影响的服务数量
- 评估 API 契约变更兼容性
- 计算风险评分并推荐审批等级

### 4.2 部署材料自动生成
- 自动调用 `impact-analysis` 技能生成影响分析报告
- 自动从追溯矩阵中提取关联的 BRS 需求清单
- 自动生成回退步骤模板

## 5. 部署后验证清单
无论风险等级，每次生产部署后均需确认：
- [ ] 健康检查端点返回正常
- [ ] 核心业务接口冒烟测试通过
- [ ] 错误率无异常升高
- [ ] 数据库连接池无异常
- [ ] 无新增 CRITICAL/ERROR 日志
- [ ] 30 分钟内无监控告警触发

## 6. 与权限体系的对接
- 本规范不替代企业现有的运维权限管理系统 (如 JumpServer/堡垒机)。
- 风险评分和审批结果作为部署申请的**材料输入**，接入企业审批系统 (如 OA/钉钉/飞书审批)。
- 技术侧确保申请材料的规范和完整性，审批权限由企业行政体系管控。
