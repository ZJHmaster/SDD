# SLA / SLO 定义框架 (Service Level Definitions)

## 术语区分

| 术语 | 全称 | 含义 |
|------|------|------|
| **SLI** | Service Level Indicator | 服务质量指标（可度量的指标） |
| **SLO** | Service Level Objective | 服务质量目标（内部承诺，如 P99 < 200ms） |
| **SLA** | Service Level Agreement | 服务等级协议（对外合同承诺，违反有赔偿） |

**关系**：SLI 是度量 → SLO 是基于 SLI 设定的目标 → SLA 是基于 SLO 对外的商业承诺。

## 核心 SLI / SLO 模板

### 可用性 (Availability)
```
SLI = 成功请求数 / 总请求数 × 100%
SLO = 99.9% (即每月允许宕机 ≈ 43 分钟)
```

| 可用性等级 | 年度允许停机 | 月度允许停机 | 适用场景 |
|-----------|-----------|-----------|---------|
| 99% (两个九) | 3.65 天 | 7.3 小时 | 内部工具 |
| 99.9% (三个九) | 8.77 小时 | 43.8 分钟 | 标准业务系统 |
| 99.95% | 4.38 小时 | 21.9 分钟 | 核心交易系统 |
| 99.99% (四个九) | 52.6 分钟 | 4.38 分钟 | 支付/金融 |

### 延迟 (Latency)
```
SLI = 请求响应时间
SLO = P50 < 50ms, P95 < 200ms, P99 < 500ms
```

### 错误率 (Error Rate)
```
SLI = 5xx 错误请求数 / 总请求数 × 100%
SLO = < 0.1%
```

### 吞吐量 (Throughput)
```
SLI = 每秒处理请求数 (QPS/TPS)
SLO = 峰值 ≥ 10,000 QPS (根据业务调整)
```

## 错误预算 (Error Budget)
- 错误预算 = 1 - SLO。
- 例如 SLO = 99.9% → 错误预算 = 0.1% → 每月允许 43 分钟不可用。
- 当错误预算耗尽时，团队应冻结新功能发布，集中精力提升可靠性。
- 这是**平衡可靠性与交付速度**的核心机制。

## 使用指南
1. 每个服务/**微服务**必须定义自己的 SLO（基于本模板调整数值）。
2. SLO 须在服务的 README 中声明。
3. 监控系统须针对 SLO 配置告警（达到预算消耗 50% / 75% / 90% 时逐级告警）。
4. 每季度回顾 SLO 目标是否需要调整。
