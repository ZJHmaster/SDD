# 团队进阶与开发培训资料 (Training Material)

这份文档供各层级研发人员翻阅，解答关于"为什么"使用这套系统体系，通过场景化来提高心智模型。

## 场景课 1：避免隐秘的灾难

**反例场景**：新人小王刚进入项目组，为了完成 "根据前台参数筛选订单" 的需求，他将传递来的字符串直接拼接进 MyBatis 或者 SQL 语句中。
**后果**：该接口被黑客扫描利用，注入了 `' OR 1=1; DROP TABLE orders; --`。公司核心营收数据被删除，遭受无法估量的损失。
**范式学习**：
- 阅读 `standards/security-standards.md` 的注入防护。
- 通过引入 Agent 中的 `/.agents/skills/error-detection` ，小王在提交的一整套流水线时就能从终端获得 AI 发出的红色严重报警。

## 场景课 2：让架构拥有生命力

**反例场景**：三年前项目的初创团队为图快，控制器（Controller/Router）里混合了入参校验、调用缓存、读写库和发送 MQ，如今这个类膨胀到了 4000 行。
**后果**：牵一发而动全身。任何对该接口的改动都不敢触碰原逻辑，只能依靠在后面不断追加 `if (xxx) { ... }` 叠甲，导致产生巨大无法重构的技术负债。
**范式学习**：
- 翻阅 `standards/architecture-patterns.md` 理解六边形/分层架构的核心：各司其职，底层向上反向解耦依赖。
- 启动 `/.agents/workflows/engineering-audit` 针对庞大的遗留 Controller，让 Agent 依据职责拆分指导给出重构和提取类的策略。

## 场景课 3：测试用例不是阻力，是保险索

**反例场景**：因工期紧张，砍掉了单元层的 AAA 测试。线上环境发现个别字段结算逻辑计算出现零点几的偏差，修复过程中反而引起更多联动报错。
**后果**：线上缺陷反复打回返工，开发陷入无意义的修补疲劳循环。
**范式学习**：
- `standards/testing-standards.md` 告诉我们越底层的逻辑（计算模型或领域核心资产）覆盖要求越严。测试能赋予开发者修改代码而不产生恐慌的底气。
- 当你需要改动高风险代码又没有测例保障时，可使用工作区下的生成技能集迅速完成用例覆盖，在被保护下修改逻辑。
