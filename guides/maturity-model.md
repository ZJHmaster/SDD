# 工程成熟度模型 (Engineering Maturity Model)

通过五个等级评估团队的工程化成熟度，帮助识别当前位置和下一步努力方向。

## 成熟度等级定义

### Level 1：初始级 (Initial)
> 🔴 混乱与英雄主义

**特征**：
- 没有统一的编码规范，每人一套风格。
- 无自动化测试或覆盖率极低。
- 部署靠手动操作。
- 故障靠最熟悉代码的"英雄"来救火。
- 没有代码评审流程。

**退出标准**：
- [ ] 团队认同需要改变。
- [ ] 指定了规范推动负责人。

---

### Level 2：可管理级 (Managed)
> 🟡 基础规范已建立

**特征**：
- 有统一的编码风格规范并在 CI 中强制。
- 有基本的 Git 分支策略和提交规范。
- 核心模块有单元测试（覆盖率 ≥ 50%）。
- 有代码评审流程（至少 1 人审批）。
- 有基本的 CI 管线（Lint + Test + Build）。

**退出标准**：
- [ ] AI Agent 接入 `/standards-check` 开始运行。
- [ ] 所有新代码经过人工或 AI 代码评审。
- [ ] 建立了 CHANGELOG 和版本管理。

---

### Level 3：定义级 (Defined)
> 🟢 流程标准化与量化

**特征**：
- 完整的 SDD `standards/` 文档已采纳并强制执行。
- 架构决策通过 ADR 记录。
- 测试覆盖率达到核心域 ≥ 80%。
- 有结构化日志和基础监控/告警。
- 定期运行 `/project-health` 和 `/engineering-audit`。
- 新人 Onboarding 流程标准化。

**退出标准**：
- [ ] 技术雷达已建立并按季度更新。
- [ ] 故障分级和响应 SLA 已明确。
- [ ] SLO 已为核心服务定义。

---

### Level 4：量化级 (Quantified)
> 🔵 数据驱动的持续度量

**特征**：
- 所有 `governance/` 规范已落地。
- 项目健康度数据持续追踪并可视化。
- 错误预算 (Error Budget) 被用于平衡交付与可靠性。
- 代码评审的平均响应时间 < 4 小时。
- 从代码提交到生产部署的 Lead Time < 1 天。
- 变更失败率 < 5%。

**退出标准**：
- [ ] DORA 四项指标全部达到 "Elite" 水平。
- [ ] 合规检查已自动化并覆盖全部清单。

---

### Level 5：优化级 (Optimizing)
> ⭐ 持续改进与创新

**特征**：
- AI Agent 深度集成于开发全流程。
- 团队主动贡献新的 Skills 和 Workflows。
- 实现了从"AI 帮助治已病"到"AI 协助查未病"。
- 定期举行技术分享和创新实验（Hack Day）。
- 规范体系反哺行业社区（开源贡献）。

---
