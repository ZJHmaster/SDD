# 代码评审检查清单 (Review Checklist)

> 此清单供人工审查及 AI Agent 自动化评审使用。每项后标注了对应的规范出处以便追溯。

## 一、功能正确性
- [ ] 代码逻辑是否完整实现了需求/工单描述的功能？
- [ ] 边界条件（空值、零值、最大值、并发）是否有处理？
- [ ] 是否与现有功能产生冲突或回归？

## 二、架构合规 → `architecture-patterns.md`
- [ ] 变更是否正确放置在对应的架构层？（领域逻辑不在Controller/Router层）
- [ ] 新增模块是否遵循 SOLID 原则？
- [ ] 是否引入了循环依赖？
- [ ] 是否遵循依赖倒置（依赖接口而非具体实现）？

## 三、编码风格 → `code-style-guide.md` + `naming-conventions.md`
- [ ] 缩进、空格、换行是否符合标准？
- [ ] 命名是否有意义且遵循约定？（无 `temp`, `data`, `obj` 等模糊词）
- [ ] 魔法数字和硬编码字符串是否已提取为常量？
- [ ] 注释是否解释了"为什么"而非"是什么"？

## 四、安全性 → `security-standards.md`
- [ ] 外部输入是否经过校验和转义？
- [ ] 是否存在硬编码的密码、密钥、Token？
- [ ] 敏感操作（删除、资金变动）是否有权限校验和审计日志？
- [ ] 日志中是否泄露了 PII/隐私数据？

## 五、异常与容错 → `error-handling.md`
- [ ] 异常是否被合理捕获和分类处理？（禁止空 `catch` 吞异常）
- [ ] 外部调用（HTTP/DB/MQ）是否有超时设置和重试/降级策略？
- [ ] 资源（连接、流、文件句柄）是否保证在 finally 中释放？

## 六、测试保障 → `testing-standards.md`
- [ ] 核心业务逻辑是否有对应的单元测试？
- [ ] 测试用例是否覆盖了正常路径和异常路径？
- [ ] 测试方法命名是否遵循条件-操作-预期格式？

## 七、性能 → `performance-standards.md`
- [ ] 是否存在 N+1 查询问题？
- [ ] 热路径（高频调用的代码段）是否必要优化？
- [ ] 大批量数据操作是否使用了分页/流式处理？

## 八、API 规范 → `api-design-guide.md`
- [ ] 接口是否遵循 RESTful 语义？
- [ ] HTTP 状态码是否准确使用？
- [ ] 是否考虑了接口的向后兼容性？

## 九、数据库 → `database-standards.md`
- [ ] Schema 变更是否以 Migration 脚本方式提交？
- [ ] 新增的查询是否有合适的索引支撑？
- [ ] 是否存在不安全的 DDL 操作（如锁全表）？

## 十、文档 → `documentation-standards.md`
- [ ] 公共API是否有完整的 JSDoc/JavaDoc 注释？
- [ ] 架构决策是否通过 ADR 记录？
- [ ] README 或变更日志是否需要更新？

## 评审结果标记
- **✅ Approved**：无阻塞问题，可以合并。
- **⚠️ Approved with minor**：有少量建议项，建议但不强制修改。
- **❌ Changes requested**：存在阻塞性问题，必须修复后重新审查。
