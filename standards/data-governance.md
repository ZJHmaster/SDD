# 数据治理与隐私合规 (Data Governance)

## 1. 数据分级分类
所有企业数据必须按照敏感度进行分级。分级结果直接影响存储、传输和访问控制策略。

| 级别 | 定义 | 示例 | 最低防护要求 |
|------|------|------|-------------|
| **L4 - 绝密** | 泄露将导致不可逆的法律或商业损失 | 核心算法源码、加密私钥 | 硬件加密模块 + 最小知晓原则 |
| **L3 - 机密** | 泄露将导致严重的合规处罚或品牌损害 | 用户身份证、银行卡号、医疗记录 | 静态加密 + 动态脱敏 + 访问审计 |
| **L2 - 内部** | 仅限公司内部使用，不对外公开 | 内部会议记录、员工薪资 | 权限控制 + 传输加密 |
| **L1 - 公开** | 可对外公开的信息 | 产品介绍、公开API文档 | 无特殊要求 |

## 2. 数据全生命周期管理
```
采集 → 传输 → 存储 → 使用 → 共享 → 归档 → 销毁
```

### 各阶段要求
- **采集**：最小化收集原则。只采集业务必需的数据，禁止过度收集。须在前端明确告知用户数据用途并获取授权。
- **传输**：全链路 TLS 1.2+ 加密。内部服务间 mTLS（双向认证）为最佳实践。
- **存储**：L3/L4 级数据必须加密存储（AES-256 或等效）。数据库级别的透明数据加密 (TDE) 是最低要求。
- **使用**：遵循"用途限制"原则——数据仅用于采集时声明的目的。
- **共享**：跨部门/跨系统共享 L3+ 数据须经数据安全委员会审批，共享接口须审计留痕。
- **归档**：超过活跃使用期的数据自动归档至冷存储，归档策略须与法规保留期对齐。
- **销毁**：数据过保留期后须不可逆销毁，并提供销毁记录。

## 3. 个人信息保护 (符合《个保法》/GDPR)
- 用户有权请求：查阅、更正、删除其个人信息（"被遗忘权"）。
- 系统须提供数据导出接口（Data Portability），允许用户以通用格式下载其全部个人数据。
- 数据泄露事件须在 72 小时内向监管机构报告（GDPR 要求）。
- 跨境数据传输须进行数据保护影响评估 (DPIA)。

## 4. 开发者实践约束
- 测试环境/开发环境禁止使用生产环境的真实用户数据。须使用脱敏后的仿真数据集。
- 代码中操作 L3+ 数据的模块须加注 `@SensitiveData` 或等效注解标记，方便 Agent 自动化扫描。
- 日志中严禁明文输出任何 L2+ 级数据项（参见 `logging-monitoring.md`）。
