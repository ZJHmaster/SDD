# 架构模式与设计标准 (Architecture Patterns)

## 1. 系统分层架构
对于企业级应用，我们统一采用**领域驱动设计(DDD)**思想启发的清晰分层架构（Clean Architecture / Hexagonal Architecture）。

### 分层定义
1. **表现层 (Presentation/API Layer)**
   - 职责：处理入站HTTP请求，输入验证，参数提取，返回响应格式化（如JSON）。
   - 限制：不允许包含任何业务逻辑。
2. **应用层 (Application/Use Case Layer)**
   - 职责：协调领域模型和基础设施服务，编排业务流（用例）。
   - 限制：不包含算法级业务逻辑，不直接依赖数据库。
3. **领域层 (Domain Layer)**
   - 职责：核心业务逻辑、实体 (Entities)、值对象 (Value Objects)、领域服务。
   - 限制：最纯净的层，不依赖其他任何外部层（零依赖架构）。
4. **基础设施层 (Infrastructure Layer)**
   - 职责：提供外部服务、数据库访问（Repository实现）、消息队列等实现。
   - 限制：仅作为底层支撑，依赖倒置（DIP）。

## 2. 核心设计原则 (SOLID)
团队所有代码必须经得起 SOLID 原则的审查：
- **S** 单一职责：一个类/模块只有一个改变的理由。
- **O** 开闭原则：对扩展开放，对修改封闭。
- **L** 里氏替换原：子类必须能够替换并作为父类型运行。
- **I** 接口隔离原理：客户端不应被迫依赖它们不使用的接口。
- **D** 依赖倒置：高层模块不依赖低层模块，两者都依赖于抽象（接口或抽象类）。

## 3. 设计模式的最佳实践
### 推荐模式：
- **工厂模式 (Factory)**：复杂对象的创建统一管理。
- **策略模式 (Strategy)**：替换庞大 `if-else` 或 `switch` 语句的首选。
- **观察者模式 (Observer / Pub-Sub)**：系统解耦、异步事件通知的基础。
- **单例模式 (Singleton)**：需谨慎使用，通常借助于依赖注入 (DI) 容器进行生命周期管理，避免硬编码。

### 反模式（严禁使用）：
- **上帝类 (God Object)**：承担过多职责的庞大类文件。
- **面条代码 (Spaghetti Code)**：无结构的流程控制，过度依赖全局变量。

## 4. 微服务架构规范 (可选)
如果采用微服务，需遵守：
- 服务应业务自治，数据库互相隔离，严禁直接跨服务读写库。
- 同步通信优先使用 gRPC 或 REST。
- 异步通信必须基于消息队列 (Kafka/RabbitMQ)，保证至少一次投递机制，接口实现幂等性。
