# 企业级测试规范 (Testing Standards)

## 1. 测试金字塔
团队实行测试金字塔策略，要求开发人员负责：
- 大量、快速执行的 **单元测试 (Unit Tests)**（占 70%）。
- 适中、涉及组件整合的 **集成测试 (Integration Tests)**（占 20%）。
- 少量、模拟真实用户行为的 **端到端测试 (E2E Tests)**（占 10%）。

## 2. 单元测试要求
- **独立性**：单元测试之间不可相互依赖，执行顺序必须可任意打乱。
- **快速执行**：不可涉及真实的数据库操作、网络请求或文件系统，一切外部依赖必须进行 Mock 隔离。
- **覆盖率**：核心领域层 (Domain Layer) 代码分支覆盖率要求达到 90% 以上。

## 3. 测试书写原则
采用 **AAA 模式** 组织测试用例代码：
1. **Arrange (准备/构造)**：初始化对象，设置 Mock 返回，准备前置数据。
2. **Act (执行)**：调用要进行测试的方法。
3. **Assert (断言)**：验证执行结果、异常状态或 Mock 方法的调用次数与参数是否符合预期。

### 命名规范
- 测试方法命名应描述"条件、操作和预期结果"。
- **范例**：`test_CalculateDiscount_WhenUserIsVip_ShouldReturnTenPercentOff`。

## 4. 持续集成与断层
集成 `.agents/workflows/standards-check.md`，测试不通过原则上禁止合并 MR (Merge Request) 或推送到生产分支。测试不仅是为了找出当前的缺陷，更是重构与防止破窗效应的护城河。
