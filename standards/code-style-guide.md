# 企业级代码风格指南 (Code Style Guide)

## 1. 核心原则
- **可读性优先**：代码不仅是给机器执行的，更是给人阅读的。
- **一致性**：整个项目和团队的代码风格必须保持高度一致。
- **自解释性**：优先通过良好的命名和结构表达意图，其次才是注释。

## 2. 格式与排版
### 2.1 缩进与换行
- 统一使用 **4 个空格**进行缩进，严禁使用 Tab。
- 单行代码长度限制在 **120个字符**以内。
- 操作符（如 `+`, `-`, `=`, `&&`）前后必须保留一个空格。
- 左大括号 `{` 不换行，右大括号 `}` 独占一行。

### 2.2 空行
- 类、接口定义之间使用两个空行。
- 方法之间使用一个空行。
- 方法内部的逻辑块之间使用一个空行，以提升可读性。

## 3. 注释规范
### 3.1 块注释 (Block Comments)
用于类、方法或复杂算法的起始处，使用 JSDoc/JavaDoc 风格：
```javascript
/**
 * 验证用户权限
 * @param {string} userId - 用户唯一标识
 * @param {string} role - 请求校验的角色
 * @returns {boolean} 拥有权限返回 true，否则返回 false
 * @throws {ValidationError} 如果参数无效
 */
function validateUserRole(userId, role) { ... }
```

### 3.2 行内代码注释 (Inline Comments)
- 仅在阐述"为什么（Why）"时使用，避免解释"是什么（What）"（代码本身应解释"是什么"）。
- 注释符后加一个空格，如 `// 这里使用二分查找以优化大型数组的性能`。

## 4. 特殊标记符号 (TODO / FIXME)
- 必须包含负责人和日期，以确保可以被追溯。
- 格式：`// TODO: [Name] [YYYY-MM-DD] [Description]`
- 格式：`// FIXME: [Name] [YYYY-MM-DD] [Description]`

## 5. 语言特定规范
### JavaScript / TypeScript
- 强制使用 `===` 和 `!==`，禁用 `==` 和 `!=`。
- 变量声明优先使用 `const`，其次 `let`，严禁使用 `var`。
- 箭头函数推荐用于匿名函数和回调。
- 优先使用 async/await 异步流控制，避免回调地狱。

### Python
- 严格遵循 **PEP 8** 规范。
- 对于所有参数和返回值必须添加类型提示 (Type Hints)。

### Java / C#
- 遵循相应语言的官方命名和格式化标准。
- 避免过长的类和复杂的方法（圈复杂度控制在 10 以内）。
