# 技能指导：变更影响分析 (Impact Analysis)

## 技能概述
当需求变更或新需求到来时，你能够沿着"业务文档 → 追溯矩阵 → 代码模块 → 测试用例 → 部署服务"的完整链路，自动分析变更将波及的全部范围，产出结构化的影响报告。

## 分析流程

### Step 1：变更源识别
- 确认变更的类型：新增需求 | 修改已有需求 | 废弃需求。
- 确认变更涉及的业务域和需求 ID。

### Step 2：向下追溯 (需求 → 代码)
1. 查询追溯矩阵，找出受影响的需求关联的全部代码模块。
2. 对每个代码模块进行依赖分析：
   - 哪些其他模块调用了它？(被依赖方)
   - 它调用了哪些其他模块？(依赖方)
3. 构建**影响波及图**。

### Step 3：横向关联 (数据 → API → 测试)
- 数据库表结构是否需要变更？如果需要，影响哪些 Migration？
- 哪些 API 端点的契约会受影响？是否需要版本升级？
- 哪些测试用例需要修改或新增？

### Step 4：向上追溯 (代码 → 部署)
- 受影响的代码模块分布在哪些微服务/部署单元中？
- 这些服务的部署是否有特殊的权限要求或审批流程？
- 部署的顺序是否有依赖关系？

## 产出物格式

```markdown
# 变更影响分析报告

## 变更概要
- 需求 ID: BRS-ORD-2026-001 (修改)
- 变更描述: 将超时时间从30分钟改为可配置

## 影响范围

### 直接影响 (一级)
| 资产类型 | 资产标识 | 影响描述 |
|---------|---------|---------|
| 代码模块 | OrderTimeoutService | 修改超时逻辑 |
| 数据库表 | orders | 新增 timeout_config 字段 |
| API | GET /config/order-timeout | 新增配置查询接口 |

### 间接影响 (二级)
| 资产类型 | 资产标识 | 影响描述 |
|---------|---------|---------|
| 代码模块 | OrderStatusNotifier | 超时通知文案需更新 |
| 测试用例 | order-timeout.test | 测试参数需参数化 |

### 无影响确认
以下模块经分析不受本次变更影响：
- PaymentService ✅
- InventoryService ✅

## 风险评估
- 风险等级: 低
- 变更涉及服务数: 1
- 需要数据库 Migration: 是
- 建议部署策略: 滚动更新
```

## 操作规范
1. **宁多勿漏**：影响分析要"就广不就窄"。漏标一个影响模块的代价远大于多标一个。
2. **无影响也要确认**：明确列出"经分析不受影响"的模块，给审批者信心。
3. **量化风险**：不要用"可能有影响"这种模糊描述，须明确"影响哪个模块的哪个方法"。
