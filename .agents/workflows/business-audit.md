# 业务文档审计工作流 (Business Audit)

定期检查业务文档是否完整、是否与代码实现保持一致、是否及时更新。

## 步骤指南

1. **文档完整性检查**
   - 扫描 `docs/biz/` 目录，验证每个业务域是否具备完整的文档集：
     - BDO (领域概览) ✅/❌
     - BGS (业务术语表) ✅/❌
     - BRL (业务规则清单) ✅/❌
     - BRS (至少一份需求规格) ✅/❌
   - 检查每份 BRS 的 YAML front matter 是否包含必填字段（docId, domain, status, traceability）。

2. **文档时效性检查**
   - 标记 `status: Approved` 但最后修改日期超过 6 个月且有新代码提交的文档 → 可能已过期。
   - 标记 `status: Draft` 超过 30 天未推进至 `Reviewing` 的文档 → 可能被遗忘。
   - 检查术语表（BGS）是否随代码中新增的领域概念同步更新。

3. **文档-代码一致性校验**
   - 对比 BRS 中 `traceability.modules` 声明的模块路径与实际文件系统：
     - 声明的模块在代码中不存在 → 文档过期。
     - 代码中新增了模块但无任何 BRS 关联 → 文档缺失。
   - 验证 BRS 中 `traceability.apis` 声明的 API 路径是否在路由配置中存在。
   - 验证 BRS 中 `traceability.tables` 声明的数据表是否在 Migration 或 Schema 中存在。

4. **业务规则覆盖检查**
   - 提取 BRL 中的所有规则编号。
   - 在代码中搜索 `@bizRule` 标记。
   - 识别有规则定义但无代码标记的"悬空规则"。
   - 识别代码中引用但 BRL 中未定义的"幽灵规则"。

5. **报告产出**
   ```markdown
   # 业务文档审计报告 - 2026-04-26

   ## 总体评分: 75/100

   | 维度 | 评分 | 说明 |
   |------|:---:|------|
   | 文档完整性 | 85 | 支付域缺少业务术语表 |
   | 文档时效性 | 70 | 3份BRS疑似过期 |
   | 文档-代码一致性 | 65 | 7个模块声明路径已变更 |
   | 业务规则覆盖 | 80 | 4条规则无代码标记 |

   ## Top 5 待修复项
   1. ...
   ```
