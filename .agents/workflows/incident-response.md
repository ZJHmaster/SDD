# 故障响应工作流 (Incident Response)

当生产环境发生故障时，提供标准化的响应流程和事后复盘模板。

## 步骤指南

### Phase 1: 检测与通报 (Detect & Alert)
- 故障通常由监控系统自动发现或用户反馈触发。
- 参照 `governance/incident-severity.md` 确定故障等级：
  - **P0 (全站不可用)**：立即电话通知值班负责人 + CTO。
  - **P1 (核心功能受损)**：15 分钟内创建故障协调群。
  - **P2 (非核心功能异常)**：1 小时内工单跟进。
  - **P3 (可感知的性能退化)**：下一工作日处理。

### Phase 2: 响应与止血 (Respond & Mitigate)
- **第一步永远是止血，而非查根因**。
- 优先考虑：
  1. 回滚到上一个稳定版本。
  2. 降级/熔断受影响的模块。
  3. 切流量到备用环境。
- AI Agent 可协助快速定位：扫描最近的部署变更、配置变更和基础设施事件。

### Phase 3: 根因分析与修复 (Root Cause Analysis)
- 故障缓解后 48 小时内完成根因分析。
- AI 代理可协助：
  - 从指定的日志片段中提取异常模式。
  - 对比故障前后的代码 Diff。
  - 审查相关模块的 `error-handling.md` 合规性。

### Phase 4: 事后复盘 (Post-Mortem)
产出标准化《故障复盘报告》，包含以下章节：
1. **故障概要**：时间线、影响范围、影响用户数。
2. **根因分析**：技术原因 + 流程原因。
3. **时间线**：从检测到完全恢复的每个关键事件节点。
4. **改进措施 (Action Items)**：每项须有明确的负责人和截止日期。
5. **经验教训**：今后如何预防类似故障。

> **核心原则**：故障复盘的目的是改进系统，不是追责个人。
