# 审计追踪要求 (Audit Trail)

## 1. 为什么需要审计追踪
在受监管行业（金融、医疗、政务），对关键业务操作的"谁、在什么时候、做了什么"必须有可追溯可举证的记录。审计追踪不仅是合规要求，也是故障排查和安全取证的重要数据来源。

## 2. 必须记录审计日志的操作
| 操作类别 | 具体操作示例 |
|---------|------------|
| **身份认证** | 登录成功/失败、注销、密码重置、MFA 验证 |
| **权限变更** | 角色分配/移除、权限策略修改 |
| **数据变更 (L3/L4)** | 创建/修改/删除敏感数据记录 |
| **配置变更** | 系统参数调整、功能开关切换 |
| **资金操作** | 支付、退款、提现、转账 |
| **数据访问 (L3/L4)** | 查看/导出敏感数据（如查询客户身份证完整号码） |
| **系统管理** | 服务启停、数据库操作、部署变更 |

## 3. 审计日志字段规范
每条审计记录必须包含：
```json
{
  "auditId": "aud_20260426_001",
  "timestamp": "2026-04-26T09:30:00.000Z",
  "actor": {
    "userId": "u_10001",
    "username": "zhangsan",
    "role": "ADMIN",
    "ip": "192.168.1.100",
    "userAgent": "Chrome/120.0"
  },
  "action": "UPDATE_USER_PROFILE",
  "resource": {
    "type": "UserProfile",
    "id": "u_20001"
  },
  "details": {
    "before": { "email": "old@example.com" },
    "after": { "email": "new@example.com" }
  },
  "result": "SUCCESS",
  "traceId": "trace_abc123"
}
```

## 4. 存储与保护要求
- 审计日志必须**只追加 (Append-Only)**，禁止修改或删除。
- 存储期限至少 **3 年**（金融行业可能要求 5-7 年）。
- 审计日志与业务日志分离存储。
- 审计日志的访问本身也需要被记录（防止内鬼篡改）。

## 5. 开发实践
- 推荐使用 AOP/中间件 在框架层面统一注入审计逻辑，而非在每个业务方法中手动编写。
- 审计日志的写入应异步化（如发送到消息队列后由独立消费者落库），避免影响主业务响应性能。
- 提供审计日志查询接口供内审团队使用，查询操作本身也需记录。
